Homework 6:

Recursive Operations with Images (100 points)

Please read important information first.

Description

In this homework, you will write a Python program to make certain calculations on pixels of images recursively.
We will use the PPM and PGM image formats for reading image files which are explained below.

PPM

The above is the content of a PPM image. The first three lines are defined as the image header. They provide an overview of the contents of the image. PPM headers consist of four entries:
  • P3 defines the image format; that is, what type of PPM (full color, ASCII encoding) image this is. For this homework, RGB images will always be P3.
  • Next comes the number of columns and the number of rows in the image. This example is a 4 pixel by 4 pixel image.
  • Finally, we have the maximum color value. This defines the scale of values possible for the color intensities. This can be any value, but a common value is 255, meaning the red, green, and blue values for a pixel can range from a value between 0 to 255. We will always use 255 for this homework.
The image body contains the actual picture information as a series of RGB (red, green, and blue) values. Each pixel of the image is a tiny, colored square.
In the PPM file, each pixel is defined by a triplet of values representing how much red, green, and blue (RGB) are present.
So, the first pixel, which has a value of 0 0 0, is black, and the last pixel, 15 15 15, is a very light shade of gray. By varying the levels of the RGB values you can come up with any color.
Note that color values must be separated by a space, but after that any additional whitespace is ignored by the image viewer. In the sample PPM above we used additional whitespace ("\t") to format the pixel values so that it is easy to understand.

PGM

This is a very similar format to PPM, the difference being that this type of file stores grayscale information. Grayscale is a range of gray shades from black to white (0 to 255).

The above is the content of a PGM image. The two notable differences are the format, which is P2 instead of P3; and the image body not having 3 values for each pixel but instead only 1.

Two of the inputs you need are taken for you, filename and operation.
filename is the name of the file you will read, and operation is a specific operation which you will apply to the image you read.
The rest of the inputs you take will depend on the value of operation. filename will always belong to files in src folder, so you can directly use them with the filename variable.
operation will always be either 1 or 2. The details of the operations are as follows:

operation == 1

You will use PGM files for this operation. In PGM files, images are represented with pixels that have a single value which represents the gray levels. This information can be represented as nested lists, where there is a single channel.
When the operation input is 1, you will apply a form of recursive region coloring to the image.
The image will contain black pixels (0 grayscale value) that separate regions. The pixels in a region will contain different grayscale values. Your task is to find the average grayscale value of all the pixels in each region and color that entire region to that region's average grayscale value. This needs to be done for all regions in a given image.

operation == 2

You will use PPM files for this operation. In PPM files, images are represented with pixels that have 3 values each which represents the 3 main colors: red (R), green (G) and blue (B). This information can be represented as nested lists, where each main color corresponds to a separate color channel, as you can see in the image below.

When the operation input is 2, you will apply the convolution operation recursively to the image.
Convolution is an operation that performs a weighted sum based on the values of a given kernel (filter) which is a 2-dimensional list of numbers, and the pixel values of a given image.
After a single weighted summation is complete, the filter moves on the image sideways by an amount named stride, and the operation is performed again until we arrive at the end of the line. After that, the filter is moved down by the amount of stride, and moved back to the beginning of the line, and the operation continues.
Meanwhile, the results of the weighted sum operations are recorded as they will form the resulting filtered image.
The image below visually demonstrates the process on an example where stride is 1:

Notice that the output dimensions may be smaller than the original image. This is because we are losing the information on the top, bottom, leftmost and the rightmost edges of the image pixel list due to the size of the filter.
You will need two additional inputs which you need to take from the user: the first is the filename of the filter which will be located under the src folder
and the second is the stride parameter, an integer which denotes how many steps the filter will move after each summation is complete. Each input will be given in a separate line.
Important details are provided below:
  • If the weighted sum exceeds the image maximum color value or becomes less than 0, you must clip it such that it is always between 0 and maximum color value.
  • If the weighted sum is not an integer, cast it to an integer instead of rounding it.
  • Filters can have different sizes, but their length and width will always be the same, like m x m where m is an odd number.
